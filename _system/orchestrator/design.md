# Orchestrator design spec
_PM OS — `_system/orchestrator/design.md`_
_Last updated: April 12, 2026_

---

## Overview

The orchestrator is the central coordination layer of the PM OS. It routes tasks to agents, manages handoffs between agents using different models, enforces human approval gates, handles crash recovery, and logs every action.

Implementation: raw Python state machine first. Introduce LangGraph after the first full product cycle runs end-to-end manually.

### Agents

| # | Agent | Stages covered |
|---|---|---|
| 1 | Research | Ideation, user research, market/competitive research |
| 2 | Product definition | Product brief, JTBD, MVP scope, metrics, feature decomposition, acceptance criteria |
| 3 | Design | UX flows, architecture, technical design, content/assets |
| 4 | Planning | Prioritization, effort estimation, sprint planning (runs post-design) |
| 5 | Dev & QA | Implementation, testing (iterative loop within agent) |
| 6 | Deployment | CI/CD, hosting, app store submission |
| 7 | Feedback | Analytics, bugs, reviews, outages → feeds back into Planning |

### Pipeline

```
Research → Product definition → Design → Planning → Dev & QA → Deployment
                                            ↑                       |
                                            |                       ↓
                                            ←——— Feedback ←————————
```

The pipeline is a loop, not a one-shot sequence. Each cycle through the loop is one of three types:

**Cycle type 1: Initial MVP build.** Full pipeline, maximum depth. Research is exploratory. Product definition from scratch. All gates active.

**Cycle type 2: Minor fixes and enhancements.** Most agents run shallow or are skipped. But any agent can be invoked deep if the specific fix demands it (e.g., a new compliance law requires deep research even in a minor fix sprint).

**Cycle type 3: Major release.** Significant new features building on an existing product. Research is targeted, not exploratory. Product definition extends rather than creates. Architecture evaluates whether existing approach still holds.

Not every cycle needs every agent. The planning agent determines which agents to activate and at what depth based on cycle type and scope. This is captured in sprint-context.md (see below).

### Agent activation

For each cycle, each agent has three possible states:

| State | Meaning |
|-------|---------|
| Full | Agent runs all sub-tasks at full depth |
| Targeted | Agent runs specific sub-tasks relevant to this cycle's scope |
| Skipped | Agent does not run this cycle |

The planning agent generates an agent activation table as part of sprint-context.md. The user reviews and adjusts at the planning gate.

### Agent architecture

An agent is NOT 1:1 with an LLM model. An agent is a Python module that owns a responsibility and calls whatever models or tools it needs per sub-task.

Each agent is defined by:
- **Inputs** — what it receives (human context, upstream agent outputs, artifact files)
- **Outputs** — what it produces (specific artifacts written to the artifact store)
- **Sub-tasks** — discrete steps to get from inputs to outputs. Each sub-task has:
  - Description
  - Execution type: LLM call, deterministic script, or tool call (web search, file I/O, git)
  - Turn type (if LLM): `single` (one prompt, one response) or `multi_turn` (conversation loop until termination condition). Most sub-tasks are single. Multi-turn is for sub-tasks requiring iterative refinement — assumption challenge, collaborative judgment points, complex design exploration.
  - Primary dimensions (if LLM) — the 2–3 eval dimensions most important for this sub-task
  - Model allocation (if LLM) — based on eval results for those dimensions

Model allocation happens at the sub-task level. A single agent may call Claude for one sub-task, Gemini for another, and a Python script for a third.

Sub-task definitions for each agent are in progress. Research agent is first.

## Context model

The system has three layers of context that serve different purposes.

### Layer 1: Persistent project context (markdown files)

These files are the shared knowledge base across all agents. They evolve slowly across the product lifecycle.

- `project-context.md` — compact always-current context (~500 words). What this product is, who it's for, core constraints, architecture choices, regulatory context. Every agent reads this first, every time. Updated only when the product direction changes — a handful of times across the entire lifecycle.
- Product artifacts (`product-brief.md`, `master-synthesis.md`, `options.md`, etc.) — read by agents as needed for specific tasks.
- Eval artifacts (`rubric.md`, `gold-set.md`) — read by the eval agent.

These files are the long-term memory of the system. They answer: what is this project, what has been decided, what are the constraints?

### Layer 2: Sprint context

`sprint-context.md` — what we are building right now. Generated by the planning agent at the start of each sprint, updated by Dev & QA as work progresses, archived when the sprint ends.

Structure:
```markdown
# Sprint context
## Cycle type: [mvp | minor | major]
## Sprint: [number]

## Scope
- [feature/fix 1]: description, acceptance criteria
- [feature/fix 2]: description, acceptance criteria

## What's already built
- [feature A]: shipped sprint 1, status
- [feature B]: phase 1 shipped, phase 2 in scope this sprint

## Deferred
- [feature C]: planned for future sprint, dependency on X

## Agent activation
| Agent | Active | Depth | Reason |
|-------|--------|-------|--------|
| Research | Yes | Targeted | New COPPA amendment needs investigation |
| Product definition | Yes | Review-only | Update brief with compliance constraint |
| Design | No | — | No UX impact |
| Planning | Yes | Full | Re-scope sprint with compliance work added |
| Dev & QA | Yes | Full | Implementation + regression testing |
| Deployment | Yes | Standard | Normal release |
| Feedback | No | — | Post-deploy, next cycle |

## Blockers and dependencies
- [blocker]: description, owner, expected resolution

## Context from previous sprints
- [relevant decisions or learnings that affect this sprint]
```

Prompt assembly includes sprint-context.md for agents 4–7 (planning, dev, deployment, feedback). Agents 1–3 (research, product definition, design) read it when activated for non-MVP cycles but not during initial MVP build.

### Layer 3: Transactional context (orchestrator state)

This is the short-term working memory. It captures what is happening right now in this run of the workflow. It exists because different agents use different models with no shared conversation history — Claude can't see what Gemini just said.

The orchestrator constructs transactional context and passes it to each agent as part of the prompt. It includes:
- What the previous agent produced
- What the human decided at the last gate (if any)
- Caveats, redirects, or constraints from the human
- What the current agent needs to do

This is the relay baton. It travels with each handoff.

### Product-level config overrides

`config-overrides.yaml` — generated by the product definition agent based on product characteristics (audience, domain, regulatory exposure, platform, quality bar). Reviewed by user at the product definition gate.

Overrides cascade into all downstream agents. Examples:
- Product involves children's data → flip all compliance judgment points to collaborative, activate accessibility testing, set content sensitivity to high
- Product is a quick prototype → reduce test coverage requirements, set most judgment points to automated, skip deployment agent monitoring depth

Config overrides include:
- Judgment point mode flips (automated ↔ collaborative) for any agent
- Operational parameters: Dev & QA iteration limit, search depth, buffer percentages
- Required sub-tasks: accessibility audit, security testing, compliance documentation
- Research agent self-configuration parameters (source recency threshold, which research sub-types to activate)

The research agent is special: it self-configures based on its inputs because it runs before product definition. All other agents receive their config from the product definition agent's overrides.

### Prompt assembly

The orchestrator constructs each agent's prompt by assembling five components:

1. **Project context** — always `project-context.md`
2. **Sprint context** — `sprint-context.md` when applicable (agents 4–7 always, agents 1–3 during non-MVP cycles)
3. **Relevant artifact files** — varies by agent and task (e.g., synthesis agent reads `master-synthesis.md`, architecture agent reads `options.md`)
4. **Task state** — the transactional context from the orchestrator (what just happened, what was decided, what to do next)
5. **Task instructions** — the specific prompt for this task, with prompt variant selected based on cycle type (mvp / minor / major)

The orchestrator is responsible for assembling 1–5 into a single prompt and dispatching it to the correct model. Config overrides from `config-overrides.yaml` influence which sub-tasks run and in what mode but are not included in the prompt itself — they're read by the agent's Python code.

### Prompt versioning

Prompts are markdown files in `_system/prompt-templates/[agent]/[task].md`. Versioning is **git-based with flat naming** — no `-v2` suffixes. Each change to a prompt is a git commit; git history is the audit trail.

**Frontmatter** carries metadata per prompt:

```markdown
---
agent: research
sub_task: scope_definition
model: claude
effort: standard       # standard | high | xhigh
turn_type: single      # single | multi_turn
---

[prompt body]
```

**SHA capture at dispatch.** To make runs reproducible, the orchestrator captures git SHAs at two granularities:

- **Per-sub-task `prompt_sha`** — recorded in the agent state file at dispatch time via `git log -1 --format=%H <path-to-prompt-file>`. Tells you exactly which version of the prompt produced each sub-task's output.
- **Run-start `run_started_at_commit`** — recorded in the orchestrator state file when a pipeline run begins. Tells you what state the entire system was in at run start.

Together these answer two reproducibility questions: "what was the system at when this run started" and "what version of this specific prompt was used for this specific sub-task call."

**Uncommitted prompt files.** Default behavior is **warn + record `dirty: true` alongside SHA** — dispatch proceeds but the agent state file flags that the prompt file had uncommitted changes. A `strict_prompt_versioning: true` config flag makes the orchestrator fail-fast (refuse to dispatch) when prompts are dirty — recommended for production / regulated runs.

### Context budget management

Each model has a context window limit. The assembled prompt (components 1–4 above) must fit within it, leaving room for the model's response.

**Context budget per model (defined in config):**

```yaml
context_budgets:
  claude:
    max_input_tokens: 180000    # conservative estimate for Opus 4.6
    reserved_for_output: 30000
    available_for_prompt: 150000
  gemini:
    max_input_tokens: 900000    # 1M context, conservative
    reserved_for_output: 100000
    available_for_prompt: 800000
  gpt:
    max_input_tokens: 120000    # GPT-5 Thinking, conservative
    reserved_for_output: 30000
    available_for_prompt: 90000
```

**Pre-dispatch check:** Before dispatching a sub-task, the orchestrator estimates the token count of the assembled prompt. If it exceeds `available_for_prompt` for the allocated model, it applies a compression strategy in this priority order:

1. **Selective inclusion** — include only sections of artifact files tagged as relevant to this sub-task, not entire files.
2. **Summarization** — for artifact files exceeding a threshold (e.g., 5K tokens), generate a compressed summary and include that instead. Log that summarization was applied so the user knows context was reduced.
3. **Fallback to larger context model** — if the sub-task's allocated model can't fit the context but another model can (e.g., Gemini's 1M window), consider switching. Log the switch.
4. **Fail and warn** — if no strategy fits, halt the sub-task and tell the user: "Context too large for any available model. Reduce artifact size or split the task."

Compression is logged in the agent state file so the user can audit whether important context was lost.

## Task states

Every task in the pipeline moves through four states:

```
queued → running → review → complete
```

- **Queued** — task defined by the orchestrator, waiting to be dispatched to an agent
- **Running** — agent is working. In autonomous mode, the agent produces output and the task moves to complete. In collaborative mode, the agent produces a draft and the task moves to review.
- **Review** — agent output is ready, waiting for human input. This state is entered in three ways: (1) the task is in collaborative mode, (2) the task has reached a scheduled gate, or (3) the user interrupted the pipeline.
- **Complete** — human approved (collaborative/gate) or autonomous task finished. Output written to artifact store. Orchestrator advances to next task.

## Interaction patterns

There are three ways a human interacts with the running system.

### 1. Scheduled gates (mandatory pauses)

Gate placement for the 7-agent pipeline (TBD — to be finalized when sub-tasks are defined, but likely candidates):
- **Post-research** — is this research sufficient? Are we looking at the right things?
- **Post-product definition** — is the product brief, MVP scope, and feature decomposition right before we design?
- **Post-design** — is the UX + architecture + technical design right before we plan and build?
- **Post-planning** — is the sprint plan realistic before we start dev?
- **Post-dev & QA** — are we ready to ship, or do we need to iterate?

Not every gate may be needed for every run. The orchestrator should allow gates to be skipped for lightweight builds.

At a gate, the orchestrator stops the pipeline, presents the relevant agent output, and waits for human input. The human can: approve and proceed, request changes (agent re-runs with feedback), or redirect the entire pipeline.

Gates are mandatory. The orchestrator will not advance past a gate without explicit human approval.

### 2. Interrupt and redirect

The user is watching output and wants to stop the pipeline, give feedback, and have the agent redo or adjust.

Mechanism: keyboard interrupt (or equivalent signal) transitions the current task from `running` to `review`. The orchestrator captures the user's feedback, appends it to the task state as a constraint, and re-runs the current agent with updated context. This is equivalent to rolling back to the last checkpoint and re-running with new inputs.

### 3. Depth toggle (autonomous vs collaborative)

Every task dispatched by the orchestrator has an interaction mode:

- **Autonomous** — agent runs, writes output, task moves to complete. No human review unless interrupted.
- **Collaborative** — agent produces a draft, presents it to the human, waits for feedback, iterates, then writes the final output. Task moves through review before complete.

The mode is set by defaults (see below) but can be overridden per-task at runtime.

### Default interaction modes

Defaults are set per task type based on irreversibility and confidence:

| Task type | Agent | Default mode | Rationale |
|-----------|-------|-------------|-----------|
| Market/competitive research | Research | Autonomous | Low irreversibility, easy to redo |
| Assumption challenging | Research | Collaborative | Human assumptions need dialogue |
| Product brief drafting | Product definition | Collaborative | High-judgment, shapes downstream decisions |
| Feature decomposition | Product definition | Collaborative | Defines what gets built |
| UX flow design | Design | Collaborative | High-judgment, user-facing |
| Architecture decision | Design | Collaborative | High-irreversibility, one-shot |
| Content/copy generation | Design | Autonomous | Low risk, easy to revise |
| Sprint planning | Planning | Collaborative | Sequencing affects everything downstream |
| Component scaffolding | Dev & QA | Autonomous | Standard patterns, low risk |
| First implementation of a feature | Dev & QA | Collaborative | Novel work, edge cases matter |
| Bug fixes on established patterns | Dev & QA | Autonomous | Constrained scope, well-defined |
| Test case generation | Dev & QA | Autonomous | Boilerplate, agent handles well |
| CI/CD setup | Deployment | Autonomous | Standard patterns |
| Analytics review | Feedback | Collaborative | Interpretation requires human judgment |

Before each run begins, the orchestrator presents the planned tasks and their default modes. The user confirms or tweaks defaults for this run. This handles the case where a normally-autonomous task needs deeper attention for a specific product (e.g., research for a compliance-heavy app needs collaborative mode even though research is normally autonomous).

## Crash recovery

### Transactional state file

The orchestrator writes a state file to disk at every state transition. Location: `_system/orchestrator/current-state.json`

The state file captures:
- Current pipeline run ID
- `run_started_at_commit` — git HEAD SHA of the pm-os repo at run start (reproducibility anchor)
- List of all tasks in the pipeline with their current state (queued/running/review/complete)
- The currently active task
- The active agent and model
- Inputs to the active task
- The last completed agent output (file path in artifact store)
- The last gate decision (if any)
- Timestamp

### Checkpoint granularity

Checkpoints happen at agent completion, not mid-agent. If the coding agent is halfway through writing a file when the system crashes, the orchestrator re-runs the coding agent with the same inputs on restart. It does not attempt to resume mid-agent.

This means:
- Checkpoint after every agent completes and writes output
- Checkpoint after every gate decision
- Checkpoint after every user interrupt and redirect
- No checkpoint during agent execution

### Restart behavior

On restart, the orchestrator reads `current-state.json` and determines where to resume:

| State at crash | Restart behavior |
|---------------|-----------------|
| Task was `queued` | Dispatch the task normally |
| Task was `running` | Re-run the agent with the same inputs (idempotent re-execution) |
| Task was `review` | Re-present the output to the user for review |
| Task was `complete` | Advance to next task |

### Idempotency

Agent tasks should be designed to be idempotent where possible. Re-running an agent with the same inputs should produce roughly equivalent output. This makes crash recovery cheap — just redo the last agent's work.

For tasks that are not naturally idempotent (e.g., implementation tasks that modify files), the state file captures enough context to reconstruct the starting conditions.

### State consistency

The system has three state stores that can get out of sync during a crash:

1. **Agent state file** (`agent-state/[agent]-[run-id].json`) — most granular. Updated async after each sub-task. Tracks exactly which sub-tasks completed, what they produced, and what judgment calls were made.
2. **Orchestrator state file** (`current-state.json`) — tracks which agent is active and its high-level status (queued/running/review/complete).
3. **Artifact store + git** — the final outputs on disk. Git commit is the last step.

**Source of truth hierarchy:** Agent state file > orchestrator state > disk/git.

A crash can create these inconsistencies:

| Scenario | Agent state says | Orchestrator says | Disk says | Resolution |
|----------|-----------------|-------------------|-----------|------------|
| Crash during sub-task | Sub-task N running, N-1 complete | Agent running | N-1 outputs on disk | Re-run sub-task N using agent state for inputs |
| Crash after sub-task write, before orchestrator update | Sub-task N complete | Agent running | N outputs on disk | Orchestrator reads agent state, updates itself, continues |
| Crash after agent complete, before git commit | Agent complete | Agent complete | Files on disk, not committed | Run git commit only |
| Crash during artifact write | Sub-task N complete | Agent running | Partial file on disk | Agent state has outputs in memory — re-write from state, then continue |

**Recovery sequence on restart:**
1. Read agent state file for the last active agent
2. Determine last completed sub-task from agent state
3. Check if outputs from that sub-task exist on disk
4. If outputs are missing or corrupted, re-write from agent state data
5. Update orchestrator state to match agent state
6. Resume from next incomplete sub-task
7. After agent completes, verify all outputs on disk, then git commit

## Error handling and retry strategy

No fallback to alternative models. When a model is unavailable, the agent pauses and waits. Token budgets on secondary subscriptions (Gemini, GPT) are too limited to absorb fallback traffic.

### Transient errors (API timeout, 429 rate limit, 500/502/503)

Retry with exponential backoff:
- Attempt 1: wait 2s
- Attempt 2: wait 4s
- Attempt 3: wait 8s
- Attempt 4: wait 16s
- Attempt 5: wait 32s

Maximum 5 retries per sub-task. If still failing after 5 retries, pause the agent and warn: "API unavailable after 5 attempts. Agent paused at sub-task N. Resume when ready."

Agent state file is current (async writes after each sub-task) — nothing is lost on pause.

### Quota/limit errors (usage cap hit, billing issue)

No retry — these won't resolve by waiting. Immediately pause and warn: "Usage limit reached for [model]. Agent paused at sub-task N. Check subscription status and resume when ready."

### Malformed response (garbage, truncated, empty)

Retry up to 2 times with the same prompt. If still bad after 2 retries, pause and present the malformed output to the user so they can diagnose. Log all attempts in agent state file.

### Resume behavior

User re-runs the orchestrator or types "resume." Orchestrator reads agent state file, finds last completed sub-task, picks up from the next one. No re-run of completed sub-tasks.

### Give up condition

If the same sub-task fails 5 times on transient errors across two separate user-initiated resume attempts (10 total tries), mark the sub-task as failed. Present options to user:
- **Skip:** proceed to next sub-task without this one's output (downstream sub-tasks may degrade)
- **Manual:** user provides the output themselves, agent continues with user-provided data
- **Abort:** stop the agent run entirely, preserve all completed work

## Agent output conventions

Each agent writes its output to a known location in the artifact store. The orchestrator uses these locations to construct the next agent's inputs.

| Agent | Output location | Format |
|-------|----------------|--------|
| Research | `research/raw/[timestamp]-[topic].md` | Structured markdown with sources, confidence levels, gaps |
| Product definition | `product/[artifact-name].md` | Product brief, feature specs, acceptance criteria |
| Design | `architecture/` and `design/` | UX flows, options analysis, decision records, data model |
| Planning | `backlog/sprint-[n].md` | Prioritized backlog, sprint plan, sequencing |
| Dev & QA | Working directory + git commit | Code files, test results, committed to repo |
| Deployment | `deployment/[environment].md` | Deploy config, CI/CD setup, monitoring |
| Feedback | `retro/[timestamp]-feedback.md` | Analytics summary, bug reports, recommendations → feeds into Planning |

The orchestrator knows where each agent writes and where the next agent reads. This is the routing logic.

## Output validation

Every sub-task output goes through validation before the agent proceeds to the next sub-task. Two types of validation, applied in order.

### Structural validation (deterministic, always runs)

Python checks that verify the output meets format and completeness requirements. These are cheap, fast, and catch obvious failures before anyone sees them.

| Check | What it verifies | Example |
|-------|-----------------|---------|
| Format compliance | Output is valid markdown/JSON, has required sections | Master synthesis has all four sections (summary, pain points, key insight, open questions) |
| Length constraints | Output meets word/token count targets | Synthesis is under 150 words as requested |
| Completeness | Output references all required inputs | Competitive analysis mentions all competitors from user-provided input |
| Schema compliance | Agent state file is valid JSON with all required fields | State file has run_id, agent, sub_tasks, judgment_calls |
| File integrity | Output file was written completely (not truncated) | File size > 0, valid UTF-8, ends with newline |

If structural validation fails, the sub-task is re-run automatically (up to 2 retries). If it fails after retries, the agent halts and presents the error to the user.

### Content validation (lightweight, configurable)

Automated checks that catch content-level errors without a full human review. These are more expensive (may require an LLM call) but catch failures that structural checks miss.

| Check | What it verifies | When to apply |
|-------|-----------------|---------------|
| Input coverage | Does the output address the key points from the input? | After synthesis, competitive analysis |
| Consistency | Does the output contradict earlier artifacts? | After any sub-task that produces a recommendation |
| Hallucination flag | Are there specific claims that don't trace back to any source? | After research sub-tasks |
| Scope compliance | Does the output stay within defined scope constraints? | After any sub-task where scope was defined in sub-task 1 |

Content validation is configurable in the agent config. For fast/prototype runs, it can be skipped entirely. For production/regulated products, all checks are active.

```yaml
validation:
  structural: always       # never skip
  content:
    input_coverage: true
    consistency: false      # expensive, enable for high-stakes
    hallucination_flag: true
    scope_compliance: true
```

### Validation in the agent state file

Each sub-task's entry in the agent state file includes a validation result:

```json
{
  "sub_task": "9_master_synthesis",
  "status": "complete",
  "prompt_sha": "abc123def456...",
  "prompt_dirty": false,
  "validation": {
    "structural": "passed",
    "content": {
      "input_coverage": "passed",
      "hallucination_flag": "passed — 2 claims flagged, both traced to sources",
      "scope_compliance": "passed"
    }
  }
}
```

If validation fails and the sub-task is re-run, both attempts are logged so the user can see what went wrong the first time.

## Observability log

Every agent action is logged to `_system/orchestrator/run-log.md` (or a structured log file) with:
- Timestamp
- Agent name
- Sub-task name and model used (agents may call different models per sub-task)
- Task description
- Input files read
- Output files written
- Duration
- Interaction mode (autonomous/collaborative)
- Gate decisions (if applicable)
- User interrupts and feedback (if applicable)

The log serves two purposes: audit trail for debugging, and recoverable context if the state file is corrupted.

## Relationship to CLAUDE.md

CLAUDE.md is a Claude Code-specific file that lives in the repo root. It provides persistent session context for the coding agent only.

When the orchestrator dispatches a task to Claude Code, Claude Code reads CLAUDE.md automatically as part of its session startup. The orchestrator's prompt assembly (project-context.md + artifacts + task state + instructions) is separate from and complementary to CLAUDE.md.

CLAUDE.md should contain:
- Repo structure overview
- Build/test/lint commands
- Code style conventions
- References to artifact files the coding agent should read for product context

CLAUDE.md is not a substitute for the orchestrator's transactional context. It provides "how to work in this codebase" context. The orchestrator provides "what to do right now" context.

## Open design questions

1. **State file format** — JSON is simple but may want to move to SQLite if the run log grows large. Start with JSON, migrate if needed.
2. **Multi-model API management** — the orchestrator needs API clients for Claude, Gemini, and GPT. Each agent may call multiple models across sub-tasks. Consider a thin adapter layer so swapping models per sub-task is a config change, not a code change.
3. **Adversarial triangulation integration** — currently a manual pattern (query all three, compare). Could be formalized as a special orchestrator mode. Not needed for v1.
4. ~~Prompt versioning~~ → Resolved: git-based + flat naming, frontmatter for metadata, per-sub-task `prompt_sha` captured at dispatch, run-start commit SHA in orchestrator state, warn-and-mark-dirty default for uncommitted prompts (fail-fast as `strict_prompt_versioning` flag). See "Prompt versioning" section.
5. **Gate placement finalization** — with 7 agents instead of 5, the original three gates need revisiting. Not every transition needs a gate. Finalize after sub-task definitions are complete.
6. **Dashboard (deferred)** — Phase 1: terminal-only. Phase 2: read-only web UI on top of state file and run log. Phase 3: active UI with gate approvals, interrupts, mode toggles. Build after first full manual product cycle.
7. **Architecture diagram update** — current diagram reflects original 5-agent structure. Needs updating for 7 agents, transactional state file, and interrupt flow.
8. **API usage/quota visibility** — do Claude Max, ChatGPT Plus, and Google AI Pro expose remaining usage/quota via API? If yes, integrate into pre-run budget estimation and mid-run monitoring. If no, track internally by logging token counts per sub-task and maintaining a running total.
9. **project-context.md update mechanism** — who updates this file as the product evolves? Consider: each agent proposes changes as an optional output, user approves at gate. Prevents staleness without unbounded growth.
10. **Multi-turn sub-tasks** — some sub-tasks (assumption challenge, collaborative judgment points) may benefit from multi-turn conversation rather than single prompt→response. `turn_type` field added to sub-task definition. Need to define termination conditions per sub-task (max turns, convergence criteria, user exit).
11. ~~API fallback strategy~~ → Resolved: no fallback. Graceful pause with retry on transient errors, immediate pause on quota errors, user-initiated resume. See "Error handling and retry strategy" section.
